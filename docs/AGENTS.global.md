## Git Commits

- Never commit to default branches (main/master/trunk). Instead branch with a
  relevant name off origin/HEAD.
- Prefer branch names prefixed `sam-`
- When asked to commit, if the branch name seems autogenerated and you haven't
  committed yet on this branch, create a new branch with a relevant name before
  committing.
- Generally prefer amending existing commits, especially for functionality
  fixes, and update the commit message to note the fix instead of creating
  follow-up fix commits.
- For `git commit` and `git commit --amend` messages:
  - use [conventional commits](https://www.conventionalcommits.org/en/v1.0.0/#summary)
  - use subject/body form with a dash-list body, a subject line of < 51
  characters, and body lines < 73 characters
  - use real newlines in commit messages instead of `\n`
  - focus more on `why` than `how` or `what`
  - wrap filenames, code snippets, variables, and identifiers in backticks
  - never add Claude/Codex/Agent Co-Authored-By lines
  - never embed literal `\n` in `-m` arguments
  - when a body is needed, write the message with real newlines using `git
    commit -F - <<'EOF' ... EOF` or a temporary message file
  - for multi-line commit messages, prefer `git commit -F - <<'EOF'` over
    shell-escaped `git commit -m` strings
  - do not try to GPG-sign commits; use `git -c
    commit.gpgsign=false commit ...` and `git -c commit.gpgsign=false commit
    --amend ...`
  - after creating or amending a commit, run `git log -1 --format=%B` and
    verify there are no literal `\n` sequences
  - if committing fails unexpectedly, fail fast and ask for help with the exact
    command, exit status, and stderr instead of papering over it

## Bash Efficiency

`set -Eeuo pipefail` is always on via BASH_ENV. Commands fail fast by default;
use subshells to isolate: `(set +e; cmd_that_might_fail) || handle_error`

Before writing Bash, map the dependency graph and encode it in one call:

- **Dependent** → `&&`: `A && B && C`
- **Independent** → `&` + `wait`: `A & B & wait`
- **Mixed** → `A && { B & C & wait; } && D`

Minimize processes: prefer multi-arg commands (`mkdir -p a b`, `wc -l f1 f2`,
`ls d1 d2`, `cat f1 f2 > out`) over separate invocations. No decorative
echo/printf for labels.

Aim for 1 call where Claude would otherwise make 3-5.

## Bash Commands — Scientific Method

Every Bash command MUST follow this pattern in a single turn:

In reasoning (not output), follow ALL steps: (1) ask a question, (2) do
background research, (3) form a hypothesis, (4) design the experiment, (5)
analyze data, (6) draw conclusions, (7) communicate.

Visible output (single turn):

1. **Before**: `🔎 [question + hypothesis in one line]`
2. **Command**: Bash tool call with concise description
3. **After**:
   - `✅ [confirmed — what we learned]`
   - `❌ [refuted — what assumption was wrong and why]`
   - `⚠️ [inconclusive — what further experiment is needed]`

When refuted, be completely honest about **why** the reasoning was wrong. Name
the incorrect assumption. Do not minimize, rationalize, or hand-wave. A refuted
hypothesis is the most valuable outcome — it means the mental model needs
updating.
